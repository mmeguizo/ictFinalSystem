- Supported chat answer families now include:
  - ticket analytics and SLA summaries
  - workload by assignee
  - approval queues
  - escalations and recent status transitions
  - MIS / ITS category breakdowns

## Important Decisions

- The AI chat is **not** open-ended NL-to-SQL.
- It uses controlled intent detection plus safe Prisma queries.
- Person-level user-directory results stay **ADMIN only**.
- Staff roles can still receive aggregate user counts and role breakdowns.
- Chat remains **read-only** for delete, deactivate, restore, and reassign actions.
- Excluded from operational chat analytics:
  - `Notification`
  - `ChatSession`
  - `ChatMessage`
  - `TicketAttachment`
  - `TicketCounter`
  - `_prisma_migrations`

## Recommended Next Steps

`chat.service.ts` now contains a large amount of admin/staff operational query logic.

Best next refactor:

- extract the operational query handlers into a dedicated helper/service
- keep `sendMessage()` as the orchestrator only
- keep role checks and formatting together per query family

### 2. Add focused validation for chat access control

Recommended manual prompt tests:

- ADMIN:
  - "how many users do we have?"
  - "show me the 3 regular users"
  - "list deactivated users"
  - "what is pending director approval?"
- Staff:
  - "show workload by staff member"
  - "which tickets are escalated?"
  - "show MIS vs ITS breakdown"
  - "show me the regular users" -> should refuse person-level list and give aggregates only
- USER:
  - analytics/report/admin queries should be denied
  - troubleshooting and own-ticket queries should still work
